## 1. Problem Understanding - Describe the problem your system solves.
System purpose: The purpose of the system is to manage user registrations for an event with a limited capacity, and automatically maintain a waitlist if the event is full
Inputs to the system:
- Event capacity
- User registration/cancellation requests
- User ID
- User status query
- Sequential operaiton execution order
Outputs produced by the system:
- Updated registration user list
- Updated waitlsit
- User status upon query
- Explicit feedback if the system is reject or succeded
Primary goal of the system:
To provide a deterministic event registraiton system which automatically manages capacity conflicts.

## 3. Functional Requirements - Describe the main behaviors the system
must perform.
FR1: The system shall register users automatically when capacity is available.
FR2: The system shall place additional users into a FIFO waitlist when capacity is full.
FR3: The system shall automatically promote the earliest waitlisted user when a registered user cancels. (Derived from Jane – predictable automated decisions)
FR4: The system shall reject duplicate registrations and provide explicit feedback. (Derived from Mo – need for clarity)
FR5: The system shall allow users to cancel their registration or waitlist position at any time.
FR6: The system shall allow users to query their current status and return a deterministic result.

## 5. Acceptance Criteria - Acceptance criteria define how system
behavior will be verified.
AC1:
Given a full event and users on the waitlist
When a registered user cancels
Then the earliest waitlisted user shall be promoted.
(Links: C1)
AC2:
Given an event at maximum capacity
When a new user registers
Then the system shall place the user in the waitlist automatically.
(Links: C2)
AC3:
Given a user submits one registration request
When the request is processed
Then the system shall produce exactly one status result.
(Links: C3)
AC4:
Given the same initial state and identical operation sequence
When operations are repeated
Then the final system state shall be identical.
(Links: C4)
AC5:
Given a duplicate registration attempt
When the system processes the request
Then the system shall reject the request and return explicit feedback.
(Links: C5, C6)
AC6:
Given a cancellation request from a non-existent user
When the system processes the request
Then the system shall return explicit failure feedback.
(Links: C5, C6)
